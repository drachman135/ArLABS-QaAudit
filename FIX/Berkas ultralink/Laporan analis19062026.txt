# Laporan Analisis Lengkap Proyek UltraLink

Laporan ini menyajikan analisis menyeluruh tentang proyek **UltraLink** (sebuah platform SaaS bio-link mirip Linktree atau Bento.me). Analisis mencakup arsitektur, dependensi, database, keamanan, serta temuan penting seperti potensi bug dan rekomendasi refactoring.

---

## 1. Struktur Project
Struktur direktori proyek menggunakan pola modern **Next.js App Router** dengan direktori `src` sebagai wadah utama kode aplikasi:

```text
ultralink/
├── .env.local             # Konfigurasi environment variables lokal (EmailJS & Cloudinary)
├── next.config.ts         # Konfigurasi runtime Next.js
├── package.json           # Definisi dependensi & skrip proyek
├── tailwind.config.ts / postcss.config.mjs # Konfigurasi styling Tailwind CSS v4
├── tsconfig.json          # Konfigurasi compiler TypeScript
├── public/                # Aset statis publik
│   ├── platforms/         # Logo-logo platform sosial media lokal (Saweria, Trakteer, dll.)
│   └── vercel.svg, next.svg, dll.
└── src/
    ├── app/               # Folder utama routing & halaman (App Router)
    │   ├── (auth)/        # Route group untuk modul autentikasi (login, register)
    │   │   ├── login/     # Halaman masuk (/login)
    │   │   └── register/  # Halaman pendaftaran (/register)
    │   ├── [username]/    # Dynamic routing untuk halaman publik pengguna
    │   ├── dashboard/     # Halaman panel kontrol pengguna (terproteksi)
    │   │   ├── links/     # Pengelolaan tautan (/dashboard/links)
    │   │   ├── profile/   # Pengelolaan profil & tema (/dashboard/profile)
    │   │   ├── layout.tsx # Layout dashboard (sidebar, info akun, logout)
    │   │   └── page.tsx   # Ringkasan analitik klik tautan
    │   ├── layout.tsx     # Root layout aplikasi
    │   ├── page.tsx       # Landing page utama
    │   └── globals.css    # Styling global dan Tailwind import
    ├── components/        # Komponen React global/reusable (FeedbackModal, RoleBadge, RoleGuard)
    ├── config/            # Konfigurasi data statis (dummyData, platforms registry)
    ├── context/           # React Context (AuthContext untuk memantau sesi pengguna)
    ├── lib/               # Inisialisasi library pihak ketiga (Firebase Client SDK)
    ├── middleware.ts      # Middleware Next.js untuk proteksi route & satpam autentikasi
    ├── services/          # Logika interaksi eksternal (Firestore, seeding dummy data)
    └── types/             # Definisi tipe TypeScript (UserProfile, UserLink, UserRole)
```

## 2. Tech Stack
Proyek ini dibangun menggunakan kombinasi teknologi modern berbasis JavaScript/TypeScript:
- **Core Framework:** Next.js (versi `16.2.9`) & React (versi `19.2.4`) menggunakan arsitektur App Router.
- **Bahasa Pemrograman:** TypeScript (`^5`) untuk pengetikan statis yang aman.
- **Styling (CSS):** Tailwind CSS v4 (`^4`) terintegrasi dengan PostCSS untuk antarmuka pengguna (UI) yang dinamis dan modern dengan performa tinggi.
- **Backend as a Service (BaaS):** Firebase (versi `12.14.0`) untuk Autentikasi, Firestore (database), dan Firebase Storage (disediakan namun tidak digunakan aktif untuk gambar).
- **Penyimpanan Gambar (Storage):** Cloudinary API untuk mengunggah foto profil secara dinamis dari sisi klien.
- **Layanan Kontak/Feedback:** EmailJS Browser (`^4.4.1`) untuk mengirim saran/keluhan pengguna langsung ke email admin tanpa server backend tambahan.
- **Manajemen Sesi:** Cookies-next (`^6.1.1`) untuk menyimpan token otorisasi Firebase di sisi klien agar bisa diakses oleh Next.js Middleware.

## 3. Database (Firestore)
Database menggunakan **Google Cloud Firestore** (NoSQL). Struktur datanya diatur ke dalam tiga koleksi (*collections*) utama:

### A. Koleksi `users`
Menyimpan profil pengguna yang terdaftar di platform.
- **Document ID:** `uid` (berasal dari Firebase Authentication)
- **Skema/Field:**
  - `username` (string): Username unik kustom untuk URL profil (lowercase).
  - `displayName` (string): Nama tampilan publik pengguna.
  - `bio` (string): Deskripsi singkat profil pengguna.
  - `avatarUrl` (string): Tautan gambar profil (di-host di Cloudinary).
  - `themeMode` (enum: `"dark" | "light"`): Pilihan tema halaman publik.
  - `accentColor` (string): Kode warna heksadesimal untuk tombol aksi.
  - `role` (enum: `"admin" | "pro" | "free"`): Level langganan/akses pengguna.
  - `createdAt` (timestamp): Waktu pendaftaran akun.
  - `updatedAt` (timestamp): Waktu pembaruan profil terakhir.

### B. Koleksi `links`
Menyimpan tautan-tautan aktif milik masing-masing pengguna.
- **Document ID:** Auto-generated oleh Firestore.
- **Skema/Field:**
  - `userId` (string): Referensi ke `uid` pemilik di koleksi `users`.
  - `title` (string): Nama label tombol link (contoh: "Instagram Portfolio").
  - `url` (string): Alamat URL tujuan link (contoh: `https://instagram.com/user`).
  - `icon` (string): Kode slug ikon platform.
  - `position` (number): Urutan posisi link di halaman publik (diurutkan naik).
  - `isActive` (boolean): Flag untuk menampilkan atau menyembunyikan link.
  - `createdAt` (timestamp): Waktu pembuatan tautan.

### C. Koleksi `analytics`
Menyimpan catatan statistik jumlah klik untuk setiap tautan.
- **Document ID:** `linkId` (sama dengan ID dokumen tautan di koleksi `links`).
- **Skema/Field:**
  - `totalClicks` (number): Jumlah akumulasi klik tautan (menggunakan operator Firestore `increment`).
  - `lastClickAt` (timestamp): Waktu terakhir tautan diklik.

## 4. Folder Penting
Direktori paling penting di dalam proyek adalah:
1. **`src/app/`**: Pusat routing Next.js. Menentukan halaman mana yang diakses oleh pengguna dan bagaimana tata letak (layouts) dirender.
2. **`src/services/`**: Berisi file `firestore.ts` yang bertindak sebagai "Data Access Layer" (semua fungsi baca/tulis data ke Firestore terkonsentrasi di sini).
3. **`src/context/`**: Berisi `AuthContext.tsx` yang membungkus aplikasi agar state autentikasi Firebase dapat diakses secara global di semua komponen klien.
4. **`src/config/`**: Folder ini berisi kamus platform (`platforms.tsx`) yang mengenali URL tujuan secara otomatis (misal: jika pengguna memasukkan url YouTube, sistem akan otomatis memasang ikon YouTube).

## 5. Routing
Aplikasi memanfaatkan fitur routing dari Next.js App Router:

| Jalur URL (Route) | File Sumber | Jenis Route | Deskripsi |
| :--- | :--- | :--- | :--- |

## 6. Controller & Model & Service

### Controller (Logika UI)
Dalam Next.js modern, logika pengontrol (controller) ditempatkan langsung di dalam komponen React (`use client` page/component). Logika ini berinteraksi dengan API/Service:
- Halaman `LinksPage` mengontrol form input tautan, memanggil fungsi Firestore, memperbarui state lokal secara instan (Optimistic UI), dan memicu alert jika terjadi kegagalan.
- Halaman `ProfilePage` bertindak sebagai pengontrol unggahan gambar ke Cloudinary dan sinkronisasi data ke Firestore.

### Model (Definisi Struktur Data)
Model diwakili oleh interface TypeScript di [index.ts](file:///c:/Users/PC/ultralink/src/types/index.ts):
- `UserProfile`: Memastikan semua properti pengguna bertipe tepat sebelum dikirim ke Firestore.
- `UserLink`: Mengatur tipe data tautan termasuk opsionalnya `id` dokumen Firestore.

### Service (Operasi Firestore & Eksternal)
- **`src/services/firestore.ts`**: Menyediakan fungsi API abstraksi seperti `getProfileByUid`, `updateProfile`, `saveUserProfile`, `getUserLinks`, `addLink`, `updateLink`, `deleteLink`, `trackLinkClick`, dan `getLinkAnalytics`.
- **`src/services/seed.ts`**: Fungsi utilitas untuk melakukan seeding database awal (membuat dokumen user dan link dummy jika dibutuhkan).

## 9. Public Renderer
Modul ini bertanggung jawab untuk merender halaman publik dinamis ketika pengunjung mengakses `ultralink.com/username`:
1. **Dynamic Metadata:** Diimplementasikan di `src/app/[username]/page.tsx` menggunakan fungsi `generateMetadata`. Ini mencari profil pengguna di Firestore dan mengembalikan judul halaman serta deskripsi kustom untuk kebutuhan SEO.
2. **Server-Side Fetching:** Halaman publik berjalan sebagai React Server Component (RSC) secara default untuk mengambil data profil (`getUserProfileByUsername`) dan tautan aktif (`getUserLinks`) dengan cepat sebelum dikirim ke browser pengunjung.
3. **Ikon Platform Otomatis:** Tautan yang terdaftar dilewatkan ke komponen `LinkButton.tsx`, yang memanggil fungsi `detectPlatform` dari `platforms.tsx`. Fungsi tersebut mencocokkan pola string URL dengan registri platform untuk menentukan ikon (SVG inline/gambar) dan warna brand tombol.

## 10. Analytics
Sistem analitik klik diimplementasikan secara tak-sinkron (asynchronous) saat pengunjung mengeklik tombol tautan:
- **Trigering:** Ketika `LinkButton` diklik, ia menjalankan `trackLinkClick(link.id)`.
- **Atomic Increment:** Di Firestore Service, total klik ditambahkan menggunakan fungsi bawaan `increment(1)` untuk menghindari masalah tabrakan data (race condition).
- **Dashboard Overview:** Akumulasi total klik dihitung di halaman `/dashboard` dengan mengambil seluruh tautan milik user, lalu memanggil `getLinkAnalytics` untuk masing-masing tautan dan menjumlahkannya secara real-time.

## 11. Storage (Cloudinary Integration)
Proyek ini memindahkan beban penyimpanan gambar dari Firebase ke **Cloudinary** demi optimasi kecepatan upload dan penanganan aset gratis yang melimpah:
- Logika upload didefinisikan dalam fungsi `uploadToCloudinary` menggunakan objek `XMLHttpRequest` mentah agar dapat memantau persentase progres unggahan (`xhr.upload.addEventListener("progress")`).
- Proses upload memanfaatkan *unsigned upload preset* (`NEXT_PUBLIC_CLOUDINARY_UPLOAD_PRESET`) yang dikirim langsung ke REST API Cloudinary (`api.cloudinary.com/v1_1/...`).

## 12. Security & API
- **Route Protection (Next.js Middleware):** `src/middleware.ts` memantau permintaan halaman. Jika pengguna mencoba mengakses `/dashboard` tanpa memiliki cookie `token`, sistem akan langsung membatalkan request dan mengalihkan pengguna ke `/login`.
- **Double-Login Prevention:** Jika pengguna yang sudah login (memiliki cookie `token`) mencoba mengakses `/login` atau `/register`, middleware akan otomatis mengalihkan mereka kembali ke `/dashboard`.
- **Firebase Auth State Sync:** Di `AuthContext.tsx`, `onAuthStateChanged` mendengarkan perubahan status login. Begitu pengguna login, token ID Firebase diambil melalui `currentUser.getIdToken()` dan disimpan ke cookie browser dengan masa kedaluwarsa 7 hari.

## 13. Potensi Bug ⚠️
Selama analisis mendalam terhadap kode sumber, ditemukan beberapa potensi bug kritis berikut:

### Bug 1: Cookie Masa Aktif vs Kedaluwarsa Token Firebase (Desinkronisasi Auth)
- **Deskripsi:** Token ID Firebase memiliki masa berlaku default hanya **1 jam**. Namun, cookie `token` disimpan dengan masa aktif **7 hari** (`maxAge: 60 * 60 * 24 * 7`).
- **Dampak:** Jika token kedaluwarsa, Next.js Middleware masih mendeteksi keberadaan cookie `token` dan mengizinkan user mengakses halaman `/dashboard`. Namun, sesampainya di dashboard, request data Firestore dari client-side SDK akan gagal karena otentikasi Firebase di balik layar sudah habis masa berlakunya atau perlu di-refresh. Hal ini menyebabkan UI kosong atau error Firestore tidak sah.

### Bug 2: Redirect Loop pada Kasus Cookie Terhapus tetapi Firebase Auth Aktif
- **Deskripsi:** Jika pengguna menghapus cookie browser secara manual tetapi sesi Firebase Auth di IndexedDB masih aktif:
  1. Middleware Next.js mendeteksi tidak ada cookie `token`, lalu mengalihkan user ke `/login`.
  2. Di halaman `/login`, `AuthContext` mendeteksi sesi Firebase Auth masih aktif (`currentUser` ada), lalu menetapkan kembali cookie `token`.
  3. Namun, karena tidak ada logika auto-redirect client-side di halaman `/login` ketika user terdeteksi sudah login, pengguna akan **tetap tertahan di halaman login** meskipun cookie `token` baru sudah dibuat. Pengguna harus me-refresh halaman secara manual agar middleware mengalihkan mereka ke dashboard.

### Bug 3: Kebocoran URL Blob pada Unggahan Foto Profil yang Gagal
- **Deskripsi:** Pada `src/app/dashboard/profile/page.tsx`, ketika user memilih file gambar, preview instan dibuat menggunakan URL blob lokal:
  ```typescript
  const localPreview = URL.createObjectURL(file);
  setProfile((prev) => ({ ...prev, avatarUrl: localPreview }));
  ```
  Jika proses unggahan ke Cloudinary selanjutnya **gagal** (mengalami error), state `avatarUrl` **tidak dikembalikan** ke URL foto profil yang lama.
- **Dampak:** Pengguna akan melihat preview foto barunya berhasil diubah di layar (karena URL blob). Jika mereka kemudian mengeklik tombol "Simpan Perubahan", URL blob lokal (`blob:http://localhost:3000/...`) akan disimpan secara permanen ke dokumen Firestore `users`. Begitu halaman dimuat ulang atau diakses oleh pengguna lain, gambar profil akan pecah (404/error) karena URL blob hanya valid di sesi browser pengunggah saat itu saja.

### Bug 4: Gambar Ikon Patreon & Ko-fi Hilang (404 Assets)
- **Deskripsi:** Registry platform di `src/config/platforms.tsx` merujuk gambar lokal `/platforms/kofi.png` dan `/platforms/patreon.png`. Namun, kedua file gambar tersebut **tidak ada** di dalam folder `public/platforms/` (hanya ada `bagibagi.png`, `cebanmatter.png`, `saweria.png`, `sociabuzz.png`, dan `trakteer.png`).
- **Dampak:** Jika pengguna memasukkan URL Ko-fi atau Patreon, tombol link di halaman publik akan menampilkan ikon yang pecah (404 Image).

### Bug 5: Batasan 5 Tautan Berlaku Juga untuk Akun PRO
- **Deskripsi:** Di `src/services/firestore.ts`, aturan batas link ditulis sebagai berikut:
  ```typescript
  if (profile?.role !== "admin") {
    const userLinks = await getUserLinks(linkData.userId);
    if (userLinks.length >= 5) {
      throw new Error("Batas maksimal 5 tautan tercapai. Hanya akun dengan role Admin...");
    }
  }
  ```
  Ini berarti pengguna dengan role `pro` akan diperlakukan sama dengan user `free` (tetap dibatasi maksimal 5 tautan) karena mereka bukan `admin`.
- **Dampak:** Skema monetisasi akun PRO menjadi sia-sia karena batas tautan tidak terangkat untuk mereka.

### Bug 6: Tautan Fallback Keras (Hardcoded) BentoLink
- **Deskripsi:** Di dalam `src/app/[username]/ActionButtons.tsx` baris ke-19, fungsi fallback URL profil jika window undefined mengembalikan:
  ```typescript
  return `https://bentolink.com/${username}`;
  ```
- **Dampak:** Nama domain salah dan mengarah ke layanan kompetitor (BentoLink) bukannya UltraLink, mengurangi nilai profesionalisme aplikasi.

## 14. Potensi Refactor ⚙️
Beberapa usulan refactoring kode untuk meningkatkan performa dan kebersihan arsitektur proyek:

1. **Hapus Kode Mati (Dead Code):**
   - Komponen `RoleGuard.tsx` sudah dibuat namun tidak pernah diimpor atau digunakan di halaman manapun di aplikasi. Komponen ini sebaiknya diintegrasikan ke halaman pengelolaan tema/aksen warna khusus PRO atau dihapus saja.
   - Fungsi `seedDummyData` di `src/services/seed.ts` tidak pernah dipanggil di mana pun.

2. **Optimasi Next.js Image untuk Cloudinary:**
   - Saat ini, foto profil dirender dengan properti `unoptimized` pada komponen `<Image />` Next.js untuk menghindari error domain tidak dikenal.
   - **Rekomendasi:** Daftarkan domain Cloudinary (`res.cloudinary.com`) ke dalam `remotePatterns` di `next.config.ts`, lalu buang atribut `unoptimized` agar Next.js dapat melakukan kompresi ukuran gambar secara otomatis demi loading halaman publik yang lebih cepat.

3. **Modernisasi Fungsi Upload Gambar:**
   - Fungsi `uploadToCloudinary` saat ini masih menggunakan `XMLHttpRequest` tradisional.
   - **Rekomendasi:** Refactor menggunakan API modern `fetch` atau Cloudinary SDK, dikombinasikan dengan state loading yang lebih sederhana untuk mengurangi boilerplate code.

4. **Sinkronisasi Otomatis Cookie Token:**
   - Alih-alih mengandalkan `onAuthStateChanged` di sisi klien untuk menulis cookie yang rawan desinkronisasi, gunakan fitur Next.js Route Handlers (API Routes) untuk menangani login/logout dan mengeset cookie HTTP-Only yang jauh lebih aman secara terpusat.
